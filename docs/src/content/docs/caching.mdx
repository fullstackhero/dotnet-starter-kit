fullstackhero uses **`HybridCache`** — the GA caching abstraction in `Microsoft.Extensions.Caching.Hybrid` — as the single caching primitive across the entire starter kit. HybridCache layers an in-process L1 on top of any distributed L2 (Redis by default), provides **built-in stampede protection**, and supports **logical tag-based invalidation** out of the box.

Modules inject `HybridCache` directly. There is no custom wrapper interface.

## Why HybridCache

Before the migration the starter kit exposed a hand-rolled `ICacheService` with two implementations (`DistributedCacheService` and a locally-built L1/L2 compositor). HybridCache replaces all of it:

| Concern | Old (custom `ICacheService`) | New (`HybridCache`) |
|---|---|---|
| L1 + L2 layering | Hand-rolled in `HybridCacheService` | Built-in |
| Stampede protection | None — concurrent misses all hit the DB | Built-in (single factory invocation per key) |
| Tag invalidation | Hand-rolled via `__tag:*` Redis sets | Built-in (`RemoveByTagAsync`) |
| Serialization | Custom STJ wrapper | Built-in STJ + pluggable `IHybridCacheSerializer<T>` |
| Maintained by | Us | Microsoft |

## The building block

`BuildingBlocks/Caching` now contains four files:

```
Caching/
├── Caching.csproj          ← references Microsoft.Extensions.Caching.Hybrid
├── CachingOptions.cs       ← bindable options (Redis conn, expirations, limits)
├── CacheKeys.cs            ← key builders + tag constants
└── Extensions.cs           ← AddHeroCaching() — registers HybridCache over Redis
```

### Registration

`AddHeroCaching(configuration)` wires L2 (Redis or in-memory fallback) then `AddHybridCache` with the options bound from the `CachingOptions` configuration section.

```csharp
builder.Services.AddHeroCaching(builder.Configuration);
```

Under the hood:

```csharp
// L2: Redis if configured, in-memory distributed cache otherwise.
if (string.IsNullOrEmpty(cacheOptions.Redis))
{
    services.AddDistributedMemoryCache();
}
else
{
    services.AddStackExchangeRedisCache(options => /* ... */);
}

// HybridCache auto-composes with whatever IDistributedCache is registered.
services.AddHybridCache(options =>
{
    options.DefaultEntryOptions = new HybridCacheEntryOptions
    {
        Expiration = cacheOptions.DefaultExpiration,            // L1 + L2 total lifetime
        LocalCacheExpiration = cacheOptions.DefaultLocalCacheExpiration, // L1 only
    };
    options.MaximumKeyLength = cacheOptions.MaximumKeyLength;
    options.MaximumPayloadBytes = cacheOptions.MaximumPayloadBytes;
});
```

### Configuration

```json
{
  "CachingOptions": {
    "Redis": "",
    "EnableSsl": false,
    "DefaultExpiration": "01:00:00",
    "DefaultLocalCacheExpiration": "00:02:00",
    "MaximumKeyLength": 1024,
    "MaximumPayloadBytes": 1048576
  }
}
```

| Property | Description | Default |
|---|---|---|
| `Redis` | Redis connection string. Empty = in-memory distributed cache. | `""` |
| `EnableSsl` | Explicit override for Redis TLS. `null` respects the connection string. | `null` |
| `DefaultExpiration` | Total cache entry lifetime (L1 + L2). | 1 hour |
| `DefaultLocalCacheExpiration` | In-process (L1) lifetime. Kept short to bound cross-node staleness. | 2 minutes |
| `MaximumKeyLength` | Maximum key length accepted by HybridCache. | 1024 |
| `MaximumPayloadBytes` | Maximum serialized payload size. Oversized entries are skipped (logged). | 1 MB |

<Aside type="note" title="Development without Redis">
If `Redis` is empty, `AddDistributedMemoryCache()` is registered as the L2 so HybridCache still works — perfect for local dev, tests, and CI.
</Aside>

## Usage

Inject `HybridCache` directly in your handlers, services, or endpoint filters. The primary API is `GetOrCreateAsync` — it handles cache lookup, stampede protection, factory invocation on miss, and write-back in one call.

```csharp
using FSH.Framework.Caching;
using Microsoft.Extensions.Caching.Hybrid;

internal sealed class UserPermissionService(
    UserManager<FshUser> userManager,
    RoleManager<FshRole> roleManager,
    IdentityDbContext db,
    HybridCache cache) : IUserPermissionService
{
    public async Task<List<string>?> GetPermissionsAsync(string userId, CancellationToken ct)
    {
        var tags = new[] { CacheKeys.Tags.Permissions, CacheKeys.Tags.User(userId) };

        return await cache.GetOrCreateAsync(
            CacheKeys.UserPermissions(userId),
            async innerCt =>
            {
                // Factory only runs on a cache miss.
                var user = await userManager.FindByIdAsync(userId).ConfigureAwait(false);
                // ... load permissions from DB ...
                return permissions;
            },
            tags: tags,
            cancellationToken: ct).ConfigureAwait(false);
    }

    public Task InvalidatePermissionCacheAsync(string userId, CancellationToken ct)
        => cache.RemoveAsync(CacheKeys.UserPermissions(userId), ct).AsTask();
}
```

### Tags and bulk invalidation

Every write can specify tags. Calling `RemoveByTagAsync` marks every entry carrying that tag as invalidated — subsequent reads treat them as misses. This is the idiomatic way to purge related entries without tracking keys manually.

```csharp
// Invalidate every theme entry for a specific tenant.
await cache.RemoveByTagAsync(CacheKeys.Tags.Tenant(tenantId), ct);

// Invalidate every permission entry for every user.
await cache.RemoveByTagAsync(CacheKeys.Tags.Permissions, ct);
```

<Aside type="tip" title="Tags are logical, not physical">
`RemoveByTagAsync` does not delete keys from Redis. It records the invalidation timestamp in HybridCache's own bookkeeping; entries tagged before that timestamp are treated as misses. Redis keys expire naturally. This is by design and is correct for read-through caches — just know that running `KEYS fsh:*` in redis-cli may show entries that are logically gone.
</Aside>

## Cache key and tag conventions

All keys and tags come from the `CacheKeys` helper in `BuildingBlocks/Caching`. Never hand-build cache keys — always use the helper so conventions stay consistent.

```csharp
public static class CacheKeys
{
    public static class Tags
    {
        public const string Permissions = "permissions";
        public const string Themes = "themes";
        public const string Idempotency = "idempotency";
        public static string Tenant(string tenantId) => $"tenant:{tenantId}";
        public static string User(string userId) => $"user:{userId}";
    }

    public static string UserPermissions(string userId)    => $"perm:u:{userId}";
    public static string TenantTheme(string tenantId)       => $"theme:t:{tenantId}";
    public const  string DefaultTheme                        = "theme:default";
    public static string IdempotencyEntry(string tenantId, string key) => $"idem:t:{tenantId}:{key}";
}
```

### Multitenancy

fullstackhero is multi-tenant by default. Cache keys and tags follow two conventions:

1. **Tenant prefix in the key** — `$"theme:t:{tenantId}"`, `$"idem:t:{tenantId}:{key}"` — gives hard isolation so a tenant can never read another tenant's entry even on key collision.
2. **Per-tenant tag** — `CacheKeys.Tags.Tenant(tenantId)` — enables a single `RemoveByTagAsync` call to purge every entry for a tenant on deactivation or deletion.

Apply both: prefix keys for isolation, tag for bulk invalidation.

## The L1 staleness tradeoff

HybridCache has **no L1 backplane**. If Node A calls `RemoveAsync` or `RemoveByTagAsync`, Node B's in-process L1 will continue serving the stale value until its `LocalCacheExpiration` elapses. Tracked at [dotnet/aspnetcore#55308](https://github.com/dotnet/aspnetcore/issues/55308).

Our starter kit accepts this tradeoff by defaulting `DefaultLocalCacheExpiration` to **2 minutes** — well under the 1-hour `Expiration` — which bounds cross-node inconsistency to at most 2 minutes for permissions, themes, and idempotency keys. If your workload needs stronger consistency:

- **Shorten `DefaultLocalCacheExpiration`** further (e.g., 30 seconds). Tradeoff: more Redis round-trips.
- **Bypass L1 for sensitive entries** using `HybridCacheEntryFlags.DisableLocalCacheRead | DisableLocalCacheWrite` on a per-entry basis.
- **Add a Redis pub/sub backplane** to call `RemoveAsync` on every node when an invalidation event is published. Not built-in — you'd wire it yourself or use FusionCache's HybridCache implementation.

<Aside type="caution" title="Probe-only reads">
HybridCache has no pure `GetAsync`. The idempotency endpoint filter uses `GetOrCreateAsync` with `HybridCacheEntryFlags.DisableUnderlyingData` to probe the cache without running a factory — a miss returns `null` and nothing is written back. Use this pattern sparingly; `GetOrCreateAsync` with a real factory is preferred.
</Aside>

## Testing

`Tests/Caching.Tests` covers the key paths:

- `CacheKeysTests` — locks in the string format of every key and tag (they're persisted to Redis, so unintentional renames would silently invalidate production entries).
- `HeroCachingRegistrationTests` — verifies `AddHeroCaching` registers `IDistributedCache` and `HybridCache`, and that `CachingOptions` binds from configuration.
- `HybridCacheBehaviorTests` — end-to-end `GetOrCreateAsync`, `SetAsync`, `RemoveAsync`, and `RemoveByTagAsync` using the in-memory distributed fallback (no Redis required).

Run them locally:

```bash
dotnet test src/Tests/Caching.Tests -c Release
```

## Migration from the old `ICacheService`

If you're upgrading an older fullstackhero project, the mapping is mechanical:

| Old | New |
|---|---|
| `ICacheService.GetOrSetAsync(key, factory, sliding, ct)` | `HybridCache.GetOrCreateAsync(key, factory, options, tags, ct)` |
| `ICacheService.GetItemAsync<T>(key, ct)` | `HybridCache.GetOrCreateAsync(key, factory, options with DisableUnderlyingData, ct)` — probe pattern |
| `ICacheService.SetItemAsync(key, value, sliding, ct)` | `HybridCache.SetAsync(key, value, options, tags, ct)` |
| `ICacheService.RemoveItemAsync(key, ct)` | `HybridCache.RemoveAsync(key, ct)` |
| `ICacheService.RemoveByTagAsync(tag, ct)` | `HybridCache.RemoveByTagAsync(tag, ct)` |

Delete the following files — they're no longer in the starter kit: `ICacheService.cs`, `CacheServiceExtensions.cs`, `DistributedCacheService.cs`, `HybridCacheService.cs`. Update all `using FSH.Framework.Caching;` call sites to also import `using Microsoft.Extensions.Caching.Hybrid;`.
