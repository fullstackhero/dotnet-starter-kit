The fullstackhero .NET Starter Kit is organized as a **modular monolith**. The codebase is split into shared framework libraries (BuildingBlocks), isolated bounded-context modules, sample host applications, and a comprehensive test suite. Every module enforces strict boundaries through a dual-project pattern, and every feature follows a consistent vertical slice layout.

## Solution Overview

The solution file `FSH.Starter.slnx` uses the modern XML-based `.slnx` format introduced in .NET 9. It contains **29 projects** organized into five logical groups:

| Group | Projects | Purpose |
|-------|----------|---------|
| **BuildingBlocks** | 11 | Shared framework libraries consumed by all modules |
| **Modules** | 8 | Bounded context modules (runtime + contracts pairs) |
| **Host** | 2 | Composition-root host applications (API, Aspire) |
| **Tests** | 5 | Unit tests, integration tests, and architecture tests |
| **Tools** | 2 | CLI tooling and EF Core migrations |

<Aside type="tip" title="Why .slnx?">
The `.slnx` format is a cleaner, XML-based replacement for the legacy `.sln` format. It is easier to read, merge, and diff in version control.
</Aside>

## Module Dual-Project Structure

Every module in the starter kit is split into exactly two projects: a **runtime** project and a **contracts** project. This separation is the foundation of module isolation.

<FileTree items={[
  { name: 'Modules.Identity/', type: 'folder', desc: 'Runtime project (internal)', children: [
    { name: 'Features/', type: 'folder', desc: 'Vertical slice feature folders' },
    { name: 'Domain/', type: 'folder', desc: 'Entities, aggregates, value objects' },
    { name: 'Data/', type: 'folder', desc: 'EF Core DbContext, configurations' },
    { name: 'Services/', type: 'folder', desc: 'Internal service implementations' },
    { name: 'Events/', type: 'folder', desc: 'Domain and integration event handlers' },
    { name: 'Authorization/', type: 'folder', desc: 'Permission and policy definitions' }
  ]},
  { name: 'Modules.Identity.Contracts/', type: 'folder', desc: 'Public API (shared)', children: [
    { name: 'v1/', type: 'folder', desc: 'Versioned commands, queries, DTOs, events' }
  ]}
]} />

**Runtime project** (`Modules.{Name}/`) contains everything internal to the module: command and query handlers, EF Core data access, domain entities, services, and event handlers. No other module should ever reference this project directly.

**Contracts project** (`Modules.{Name}.Contracts/`) is the public API surface. It defines the commands, queries, response DTOs, integration events, and service interfaces that other modules are allowed to consume.

<Aside type="note" title="Module Boundary Rule">
A module **must not** reference another module's runtime project - only its Contracts project. This ensures modules remain loosely coupled and can evolve independently. This rule is automatically enforced by architecture tests in `Architecture.Tests/`.
</Aside>

## Feature Folder Layout

Inside every runtime module, features follow the **Vertical Slice Architecture** pattern. Each feature is a self-contained folder that groups its endpoint, handler, and validator together - no scattering code across layers.

The convention is:

<FileTree items={[
  { name: 'Features/', type: 'folder', children: [
    { name: 'v1/', type: 'folder', desc: 'API version', children: [
      { name: '{Area}/', type: 'folder', desc: 'Logical area (e.g., Users, Roles)', children: [
        { name: '{FeatureName}/', type: 'folder', desc: 'One folder per feature', children: [
          { name: '{FeatureName}Endpoint.cs', desc: 'Minimal API endpoint definition' },
          { name: '{FeatureName}CommandHandler.cs', desc: 'CQRS command or query handler' },
          { name: '{FeatureName}CommandValidator.cs', desc: 'FluentValidation validator' }
        ]}
      ]}
    ]}
  ]}
]} />

Here is a real example from the Identity module:

<FileTree items={[
  { name: 'Features/v1/Users/RegisterUser/', type: 'folder', children: [
    { name: 'RegisterUserEndpoint.cs', desc: 'POST /api/v1/identity/register' },
    { name: 'RegisterUserCommandHandler.cs', desc: 'Handles the RegisterUserCommand' },
    { name: 'RegisterUserCommandValidator.cs', desc: 'Validates registration input' }
  ]}
]} />

The corresponding command and response types live in the Contracts project:

<FileTree items={[
  { name: 'Modules.Identity.Contracts/v1/Users/RegisterUser/', type: 'folder', children: [
    { name: 'RegisterUserCommand.cs', desc: 'Command definition (ICommand)' },
    { name: 'RegisterUserResponse.cs', desc: 'Response DTO' }
  ]}
]} />

This pattern keeps every file related to a feature co-located. When you need to understand or modify user registration, everything is in one place.

## Dependency Hierarchy

The projects follow a strict dependency hierarchy that flows in one direction - from foundational libraries up through modules to host applications.

<FileTree items={[
  { name: 'Core', type: 'folder', desc: 'No dependencies - DDD primitives, exceptions, CQRS interfaces', children: [
    { name: 'Shared', type: 'folder', desc: 'Constants, DTOs, permission contracts', children: [
      { name: 'Persistence', type: 'folder', desc: 'EF Core abstractions, specifications, outbox/inbox' },
      { name: 'Eventing.Abstractions', type: 'folder', desc: 'Event contracts and interfaces', children: [
        { name: 'Eventing', type: 'folder', desc: 'Domain event dispatching, integration events' }
      ]},
      { name: 'Caching', type: 'folder', desc: 'Distributed and hybrid cache (Redis)' },
      { name: 'Jobs', type: 'folder', desc: 'Hangfire background job infrastructure' },
      { name: 'Mailing', type: 'folder', desc: 'Email abstraction layer' },
      { name: 'Storage', type: 'folder', desc: 'File storage abstraction' },
      { name: 'Web', type: 'folder', desc: 'Host infrastructure, middleware, module system', children: [
        { name: 'Module Contracts', type: 'folder', desc: 'Public API surface for each module', children: [
          { name: 'Module Runtimes', type: 'folder', desc: 'Internal implementation per module', children: [
            { name: 'Host (host apps)', type: 'folder', desc: 'API, Aspire - compose modules into runnable hosts' }
          ]}
        ]}
      ]}
    ]}
  ]}
]} />

- **Core** sits at the bottom with zero external project dependencies. It defines the DDD primitives, exception types, and CQRS interfaces that everything else builds on.
- **Shared** adds cross-cutting constants, DTOs, and permission contracts.
- **Infrastructure libraries** (Persistence, Caching, Eventing, Jobs, Mailing, Storage) each depend on Core and/or Shared but not on each other.
- **Web** ties the infrastructure together and provides the module loading system.
- **Modules** consume BuildingBlocks and other modules' Contracts projects - never their runtime projects.
- **Host** applications sit at the top, composing modules into runnable hosts.

<Aside type="tip" title="Enforced by Architecture Tests">
The dependency hierarchy is enforced by architecture tests in `Architecture.Tests/`. Any violation - such as a module referencing another module's runtime project or a BuildingBlock depending on a module - will fail the build.
</Aside>

## Configuration Files

The `src/` directory contains several configuration files that govern how the entire solution is built and styled.

| File | Purpose |
|------|---------|
| `Directory.Build.props` | Shared MSBuild properties applied to every project - target framework (`net10.0`), nullable reference types, implicit usings, analyzers, and common metadata. |
| `Directory.Packages.props` | Central package management. Every NuGet package version is declared here in a single place, so individual `.csproj` files reference packages without specifying versions. |
| `.editorconfig` | Coding style rules enforced by the IDE and analyzers - indentation, namespace style, naming conventions, and severity levels. |
| `FSH.Starter.slnx` | The solution file that ties all 29 projects together into the logical groups shown above. |

<Aside type="warning" title="BuildingBlocks Are Protected">
The `BuildingBlocks/` directory contains shared framework libraries consumed by every module in the solution. Changes here have a **wide blast radius** and can break multiple modules simultaneously. Do not modify BuildingBlocks without explicit approval and thorough testing across all dependent modules.
</Aside>
