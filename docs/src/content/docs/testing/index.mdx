fullstackhero ships with around **900 `[Fact]` + `[Theory]` tests** that run green on every build. Three layers: unit tests verify domain invariants and handler logic with no infrastructure; integration tests spin up Postgres, Redis, and MinIO via Testcontainers and exercise the API end-to-end through `Microsoft.AspNetCore.Mvc.Testing`; architecture tests use NetArchTest to enforce module boundaries, contracts purity, and naming conventions as compiler-level rules.

<Callout type="tip" title="The test suite is the contract">
Every architectural rule in the kit — "modules never reference each other's runtime", "every handler has a validator", "no controller may bypass `[RequirePermission]`" — is a NetArchTest assertion. Break the rule, fail the build. Read `src/Tests/Architecture.Tests/` to see exactly what's enforced.
</Callout>

## The test stack

| Layer | Library | Version | Used for |
|---|---|---|---|
| Test runner | xUnit | 2.x | All test projects |
| Assertions | Shouldly | 4.x | Readable assertions (`result.ShouldBe(...)`, `.ShouldThrow<>`, `.ShouldSatisfyAllConditions(...)`) |
| Mocking | NSubstitute | 5.x | Service interfaces and dependency stand-ins |
| Test data | AutoFixture | 4.x | Generated DTOs / random fixtures |
| Integration host | `Microsoft.AspNetCore.Mvc.Testing` | 10.0.8 | `WebApplicationFactory<T>` for in-process API tests |
| Containers | Testcontainers | 4.11 | Real PostgreSQL 17, Redis 7, MinIO instances per test class |
| Architecture | NetArchTest.Rules | 1.3 | Module boundary + naming + dependency rules |

Everything is pinned in `src/Directory.Packages.props` (central package management).

## Unit tests

Naming convention: `MethodName_Should_ExpectedBehavior_When_Condition`. Layout: Arrange → Act → Assert grouped under `#region` blocks (Happy Path / Exception / Edge Cases).

```csharp
// src/Tests/Identity.Tests/Features/v1/Users/RegisterUser/RegisterUserCommandHandlerTests.cs
public sealed class RegisterUserCommandHandlerTests
{
    private readonly Fixture _fixture = new();
    private readonly IUserRegistrationService _registration = Substitute.For<IUserRegistrationService>();

    [Fact]
    public async Task Handle_Should_DelegateToRegistrationService_When_CommandIsValid()
    {
        // Arrange
        var command = _fixture.Build<RegisterUserCommand>()
            .With(c => c.Email, "valid@example.com")
            .With(c => c.Password, "Pa55w0rd!Strong#Enough")
            .Create();
        var expected = new RegisterUserResponse(Guid.NewGuid(), command.Email);
        _registration.RegisterAsync(command, Arg.Any<CancellationToken>())
                     .Returns(ValueTask.FromResult(expected));

        // Act
        var sut = new RegisterUserCommandHandler(_registration);
        var actual = await sut.Handle(command, CancellationToken.None);

        // Assert
        actual.UserId.ShouldBe(expected.UserId);
        await _registration.Received(1).RegisterAsync(command, Arg.Any<CancellationToken>());
    }
}
```

Three ground rules:

- **No `Assert.True(condition, "message")`.** Use Shouldly's `.ShouldBe()`, `.ShouldNotBeNull()`, `.ShouldThrow<T>()` so failure messages carry the actual + expected without your help.
- **No real DI container in unit tests.** Construct the SUT directly with NSubstitute substitutes for dependencies.
- **No file system, no network, no DbContext.** If you need any of those, it's an integration test.

## Integration tests

Integration tests boot the API in-process through `WebApplicationFactory<TStartup>`, wired to Testcontainers Postgres + Redis + MinIO. The same containers serve every test class in the project; startup cost is paid once.

```csharp
// src/Tests/Integration.Tests/Tests/Catalog/ProductsEndpointTests.cs
public sealed class ProductsEndpointTests(FshWebApplicationFactory factory) : IClassFixture<FshWebApplicationFactory>
{
    private readonly HttpClient _client = factory.CreateAuthenticatedClient(tenant: "acme");

    [Fact]
    public async Task POST_products_Should_ReturnCreated_And_PersistProduct_When_PayloadIsValid()
    {
        var command = new CreateProductCommand(
            Sku: "TEST-001", Name: "Test product", Description: null,
            BrandId: Seed.Brands.Default.Id, CategoryId: Seed.Categories.Default.Id,
            PriceAmount: 19.99m, PriceCurrency: "USD", Stock: 100m);

        var resp = await _client.PostAsJsonAsync("/api/v1/catalog/products", command);

        resp.StatusCode.ShouldBe(HttpStatusCode.Created);
        var dto = await resp.Content.ReadFromJsonAsync<ProductResponse>();
        dto!.Sku.ShouldBe("TEST-001");
    }
}
```

The kit's `FshWebApplicationFactory.CreateAuthenticatedClient(tenant)` does three things automatically:

1. Mints a JWT for a seeded test user with all permissions, signed by the test host's signing key.
2. Sets the `Authorization: Bearer ...` header.
3. Sets the `tenant: acme` header so Finbuckle resolves the right tenant context.

Adjacent helpers — `CreateAnonymousClient`, `CreateClientWithoutPermissions`, `WithImpersonation` — cover the auth-edge cases.

The Catalog module has the densest integration tests (six files, ~70 KB of test code) and is the best place to start reading.

## Architecture tests

NetArchTest checks compile-time rules. Two example rules:

```csharp
// Architecture.Tests/Modules/ModuleBoundaryTests.cs
[Fact]
public void Modules_runtime_should_not_reference_other_modules_runtime()
{
    var modules = new[] { "Modules.Identity", "Modules.Catalog", "Modules.Chat", /* ... */ };
    foreach (var (asm, others) in modules.Select(m => (Assembly.Load(m), modules.Where(o => o != m))))
    {
        var result = Types.InAssembly(asm)
            .Should()
            .NotHaveDependencyOnAny(others.ToArray())
            .GetResult();
        result.IsSuccessful.ShouldBeTrue($"Forbidden cross-module reference: {string.Join(", ", result.FailingTypeNames ?? [])}");
    }
}

// Architecture.Tests/Endpoints/EndpointConventionTests.cs
[Fact]
public void Every_endpoint_method_must_be_static_extension_on_IEndpointRouteBuilder()
{
    var result = Types.InAssemblies(/* runtime modules */)
        .That()
        .HaveNameEndingWith("Endpoint")
        .Should()
        .BeSealed()
        .And().HaveMethods(/* MapXxxEndpoint */)
        .GetResult();
    result.IsSuccessful.ShouldBeTrue();
}
```

The kit's architecture rules cover:

- **Module boundaries** — no module runtime references another module runtime (`ModuleBoundaryTests`).
- **Contracts purity** — contracts assemblies don't reference EF Core, Mediator (runtime), or other modules (`ContractsPurityTests`).
- **Building blocks independence** — building blocks don't reference modules (`BuildingBlocksIndependenceTests`).
- **No circular references** between projects (`CircularReferenceTests`).
- **Handler / validator pairing** — every command/query has a matching validator if one is expected (`HandlerValidatorPairingTests`).
- **Domain entity rules** — aggregates are sealed, raise events via the protected helper, never have public setters (`DomainEntityTests`).
- **API versioning** — every endpoint belongs to a versioned route group (`ApiVersioningTests`).
- **Feature folder convention** — every folder under `Features/` starts with `v{N}` (`FeatureArchitectureTests`).
- **Host architecture** — the host project doesn't reference module internals (`HostArchitectureTests`).

There are **48 architecture tests** total. Read the source — it's the most concise statement of the kit's rules.

## Running tests

```bash
# Everything
dotnet test src/FSH.Starter.slnx

# A single module's unit tests
dotnet test src/Tests/Identity.Tests/

# Integration tests only (Docker must be running for Testcontainers)
dotnet test src/Tests/Integration.Tests/

# A single test by name
dotnet test --filter "FullyQualifiedName~ProductsEndpointTests"

# Architecture tests only — fast, no containers
dotnet test src/Tests/Architecture.Tests/
```

CI runs all three layers on every push. Integration tests are the slowest (~30-60 seconds depending on cold/warm container start); unit + architecture tests are sub-second per project.

## How to add a new test

### A unit test for a new handler

1. Create the test file mirroring the handler's path: `src/Tests/{Module}.Tests/Features/v1/{Area}/{Feature}/{Feature}CommandHandlerTests.cs`.
2. Inherit no class — xUnit discovers `[Fact]`-attributed methods directly.
3. Substitute every dependency with NSubstitute.
4. Use AutoFixture for boilerplate DTOs; override fields you actually care about with `.With(...)`.

### An integration test for a new endpoint

1. Add a file under `src/Tests/Integration.Tests/Tests/{Module}/{EndpointName}Tests.cs`.
2. Inherit `IClassFixture<FshWebApplicationFactory>`.
3. Call `factory.CreateAuthenticatedClient(tenant: "acme")` (the seeded test tenant).
4. Use one of the seed helpers (`Seed.Brands.Default`, etc.) to satisfy foreign keys.
5. Assert against `HttpStatusCode`, deserialise the body with `ReadFromJsonAsync<>()`, then assert with Shouldly.

### An architecture rule

Add to `src/Tests/Architecture.Tests/` with a descriptive `[Fact]`. Run locally; it should be near-instant. Document the rule's *why* in the test's XML doc comment — that comment is the readable specification, the assertion is the enforcement.

## What the test suite does NOT cover

Honest disclaimers:

- **Load testing.** The kit has no built-in stress test scaffolding. For load tests against your fork, [k6](https://k6.io/) or [NBomber](https://nbomber.com/) integrate well with the kit's OpenTelemetry exporter.
- **Visual regression on the React apps.** `clients/admin` + `clients/dashboard` have basic unit tests via Vitest but no Storybook / Chromatic / Playwright suite yet.
- **Chaos engineering.** Cluster failure modes, network partitions, container restarts — not covered in the in-repo tests. Run your own chaos suite against your deployed environment.

For day-one shipping, the existing 900 tests carry you a long way. Add to them as you add features.
