The Blazor.UI block is currently a **placeholder**. The directory `src/BuildingBlocks/Blazor.UI/` exists to reserve the name in the project structure, but there is no `.csproj` file shipping in v10 and no source code inside the directory.

<Callout type="warning" title="Not yet implemented">
v10 ships two production React + Vite frontends — `clients/admin` (the operator console) and `clients/dashboard` (the tenant-facing app). If you want Blazor, this is the slot where a shared Blazor Server / WebAssembly component library will eventually land. For now the slot is empty.
</Callout>

## What's planned (subject to change)

If the kit grows a Blazor frontend story, this block is where it'll live:

- Shared form primitives (text, select, date, file picker, money input)
- Shared layout primitives (page header, sidebar, breadcrumbs)
- Shared data-table component with sort + paginate + filter
- Authentication primitives bound to the kit's JWT bearer flow
- Multi-tenant theme primitives reading `TenantTheme` from the Multitenancy module
- Mobile-friendly modals / sheets / toasts

This block deliberately doesn't conflict with the React frontends in `clients/`. They serve different audiences (React for the production admin/dashboard apps, Blazor for teams that want a single .NET stack end-to-end).

## What's actually shipping for frontend in v10

| Surface | Stack | Source |
|---|---|---|
| Admin console (operators, support) | React 19 + Vite + TypeScript 5.7 | `clients/admin/` |
| Tenant dashboard (end users) | React 19 + Vite + TypeScript 5.7 | `clients/dashboard/` |

Both are wired against the API's OpenAPI schema, both have permission-driven UI, both ship with realtime SignalR support for chat + notifications.

## If you want to implement Blazor.UI

The block is reserved but unimplemented. You can:

1. Create `src/BuildingBlocks/Blazor.UI/Blazor.UI.csproj` targeting `net10.0` with `Microsoft.NET.Sdk.Razor`.
2. Add it to the solution (`src/FSH.Starter.slnx`).
3. Reference it from `src/Host/FSH.Starter.Blazor/` (if the Blazor host exists) or from a new Blazor host project.
4. Start adding shared components — modeled on what the React apps render today, so the two stacks have feature parity.

The maintainers will eventually formalise this; the most likely path is "ship a small set of primitives that mirror the editorial design language used elsewhere in the kit."
